# قواعد عمل المساعد الذكي على هذا المشروع

هذه القواعد **إلزامية** لأي تعديل يقوم به المساعد الذكي (Lovable / أي AI agent) على هذا المشروع. الهدف: حماية البنية، منع التكرار، ومنع كسر القواعد المتفق عليها.

## قبل كتابة أي كود

يجب على المساعد:

1. **قراءة ملفات `/docs` الأربعة**:
   - `docs/PROJECT_STRUCTURE.md`
   - `docs/CODING_CONVENTIONS.md`
   - `docs/DECISIONS.md`
   - `docs/AI_WORKFLOW_RULES.md`
2. **البحث في الكود الموجود** عن أي شيء له علاقة بالمهمة (`rg "<keyword>" src supabase`).
3. **عرض الملفات** التي يعتزم تعديلها أو إنشاءها.
4. **شرح سبب** كل تعديل بسطر أو سطرين.
5. **الانتظار للموافقة** قبل تنفيذ أي تعديل ذي أثر معماري أو على قاعدة البيانات.

> التعديلات الصغيرة المعزولة (إصلاح نص، تنسيق، تعديل لون) يمكن تنفيذها مباشرة، مع الإبلاغ بعدها.

## ممنوعات صريحة

- ❌ إنشاء ملف جديد إذا كان يوجد ملف يمكن تعديله بنفس المسؤولية.
- ❌ تكرار دالة أو منطق موجود بأسماء مختلفة.
- ❌ استدعاء `supabase.*` مباشرة من داخل `src/components/*` أو `src/pages/*`. يجب المرور عبر `src/services/supabase/`.
- ❌ وضع أي مفتاح/سرّ داخل `src/` أو `.env` أو ملفات الكود.
- ❌ تنفيذ منطق RAG / embeddings / استدعاء Gemini داخل الواجهة.
- ❌ تعديل أي من:
  - `src/integrations/supabase/client.ts`
  - `src/integrations/supabase/types.ts`
  - `.env`
  - مفاتيح المشروع في `supabase/config.toml`
- ❌ إجراء refactor كبير لم يُطلب صراحةً.
- ❌ نقل ملفات بين المجلدات دون موافقة.
- ❌ تغيير الثيم/الألوان/الخط بدون طلب.
- ❌ كتم الأخطاء (`catch {}` بدون تعليق).

## بعد كل تعديل

يجب على المساعد عرض تقرير قصير يحوي:

1. **الملفات التي تم تعديلها أو إنشاؤها** (قائمة واضحة).
2. **سبب كل تعديل** بسطر واحد لكل ملف.
3. **المخاطر المحتملة** (مثلاً: قد يؤثر على الكاش، يحتاج إعادة تشغيل، يغيّر سلوك RAG…).
4. **كيفية الرجوع عن التعديل** (مثل: حذف الملف الجديد، استرجاع نسخة سابقة، تشغيل migration عكسي).

### قالب التقرير

```text
الملفات المعدّلة:
- src/features/chat/services/chatApi.ts — السبب: …
- supabase/functions/chat/index.ts — السبب: …

المخاطر:
- …

الرجوع:
- …
```

## قواعد خاصة بـ Supabase

- أي تغيير على السكيمة → عبر **migration** فقط، لا تعديل مباشر للجداول.
- أي RPC جديد → SQL داخل migration، ثم استهلاكه عبر `services/supabase/`.
- جدول جديد → فعِّل RLS وضع policies مناسبة في نفس الـ migration.
- أي سرّ جديد → اطلبه من المستخدم عبر أداة الأسرار، لا تخمنه.

## قواعد خاصة بـ Edge Functions

- المنطق الثقيل (RAG, AI calls, crawling, embeddings) **حصراً هنا**.
- المشترك بين functions → في `supabase/functions/_shared/`.
- لا تعطّل `verify_jwt` بدون سبب موثَّق.
- اقرأ المفاتيح من `Deno.env.get(...)` فقط.

## الترتيب الموصى به للمهام

1. فهم الطلب وتحديد الميزة (`chat`, `admin`, `public-feedback`, `export`).
2. تحديد الطبقة (UI / hook / service / edge function / migration).
3. مراجعة `PROJECT_STRUCTURE.md` لتأكيد المسار الصحيح.
4. مراجعة `CODING_CONVENTIONS.md` لاتباع الأسلوب.
5. التنفيذ بأقل عدد ملفات وأقل سطور كافية.
6. التحقق (build/console/network) ثم تقرير ما بعد التعديل.

## في حال التعارض

إذا تعارض طلب المستخدم مع هذه القواعد:
- نبّه المستخدم إلى التعارض ومكانه (أي قاعدة).
- اقترح بديلاً يحترم القواعد.
- لا تنفّذ ما يخالف القواعد إلا بموافقة صريحة وتسجيل الاستثناء في `DECISIONS.md`.
